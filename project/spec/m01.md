# M01: Security Data Substrate

## Objective

Build the normalized ingestion, retention, replay, and derived-state substrate needed for the first product workflows.

## User-Visible Outcome

The system can ingest a fixed source pack through both pull and push paths, retain raw events for a short hot window, preserve long-lived analytical state in the customer account, and replay data safely without coupling processing to the stateless API tier.

## Required Capabilities

- source adapters for `Okta`, `Google Workspace`, `GitHub`, and `Snowflake`
- separate worker roles for collection, ingestion, normalization, and detection materialization
- a stateless API tier that serves read models without running ingest or scoring inline
- an authenticated ingestion endpoint suitable for high-volume pushed telemetry
- a versioned normalized event schema
- append-only raw event batch storage in the frozen lake layout contract
- object-storage-backed normalized analytical event batches in columnar form
- asynchronous queue semantics for processing coordination and replay
- manifest and catalog metadata supporting idempotency and replay
- a versioned object-pointer contract linking evidence, retained events, and future hot indexes
- a versioned Postgres hot event index contract for event list, detail, and detection-linked reads
- a versioned identity contract covering tenants, integrations, events, entities, and alias bindings
- derived-state and detection materialization jobs
- event indexes linking retained events, detections, and evidence retrieval
- retention enforcement for raw objects
- a synthetic-first fixture path that exercises the fixed source pack without changing the stable contracts

## Acceptance Criteria

- end-to-end ingest exists for `Okta`, `Google Workspace`, `GitHub`, and `Snowflake`
- both scheduled collectors and push ingestion can land data into the same raw intake contract
- the API can continue serving read traffic without performing ingest or detection work inline
- duplicate and late-arriving events are handled safely
- raw and normalized batches land under the versioned object-key and manifest contract defined in `/Users/inowland/Development/seccloud/project/spec/lake-contract.md`
- hot event index rows and detection-event edges land under the versioned Postgres contract defined in `/Users/inowland/Development/seccloud/project/spec/hot-event-index-contract.md`
- event and entity references obey the versioned identity contract defined in `/Users/inowland/Development/seccloud/project/spec/identity-contract.md`
- raw retention can be enforced without invalidating permitted derived state
- derived state can be rebuilt from retained artifacts and manifests
- detections remain linked to retained event records and evidence pointers after raw retention within the approved contract
